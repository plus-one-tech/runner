## 改善メモ（runner）

### 目的

`.run` ファイルに OS ごとのスクリプトをまとめて記述できるようにし、利用者は OS を意識せず `runner file.run` を実行するだけで、その環境に応じた処理を実行できるようにする。

---

### 現状の課題

* `.run` の仕様が曖昧で、`#bash` 型と `#script` 型の考え方が混在している
* `#script` が特別扱いされず、runtime 名として解釈されてしまう
* `@windows` `@linux` `@macos` などの OS 制御構文が未実装
* Windows で `pwsh` 実行時、一時ファイル拡張子が `.ps1` でないと失敗する
* `runner` の再帰呼び出し時に、PowerShell からコマンド名 `runner` が見つからない場合がある
* dry-run の表示が外側 1 回分だけで、中身の確認には向いていない
* Git Bash 対応まで含めると、パス変換や挙動差異で話が複雑になる

---

### 採用したい方針

* `.run` は runner 専用のスクリプトコンテナとする
* `.run` の先頭は `#script` とする
* 1つの `.run` に `@windows` `@linux` `@macos` の各ブロックを書けるようにする
* runner は現在の OS に該当するブロックのみを抽出して実行する
* 各 OS ブロックの中身は、その OS 向けの素のスクリプトをそのまま書く
* 利用者は `runner exec.run` だけを実行すればよい
* 目的は「同じスクリプトを書くこと」ではなく「同じ結果を得ること」とする

---

### 想定する基本仕様

* 先頭行は `#script`
* OS ブロックは `@windows` `@linux` `@macos`
* 終端は `@end`
* runner は現在 OS に対応するブロックだけを抽出する
* Windows は抽出結果を `.ps1` の一時ファイルに書いて `pwsh` で実行する
* Linux / macOS は将来 `sh` または `bash` 実行を想定する
* Git Bash は当面対象外とする

---

### 仕様補足（重要）

* 現在 OS に対応するブロックが存在しない場合はエラーとする

  ```
  [runner] no script for current os: windows
  ```
* 同一 OS ブロックが複数存在する場合はエラーとする
* OS ブロックが1つも存在しない場合はエラーとする
* 未知の `@xxx` は無視する（dry-run では警告表示）
* コメント行は `#` で始まる行とする（1行目の `#script` を除く）
* 空行は無視する
* スクリプト本文のインデントや空白はそのまま保持する
* 改行コードは入力ファイルの内容をそのまま維持する

---

### 今回のスコープに入れるもの

* `#script` の正式対応
* `@windows` `@linux` `@macos` `@end` の実装
* 対象 OS ブロックの抽出
* Windows / PowerShell 向けの一時ファイル拡張子 `.ps1` 対応
* dry-run で抽出後スクリプトや実行コマンドを確認できるようにする
* テストデータ整備

---

### 今回のスコープから外すもの

* Git Bash 対応
* `-ignore` の実装
* 変数
* ループ
* runner 独自の if 構文
* 共通ブロック（`@common` など）
* 複数 OS を 1 行でまとめる記法
* FreeBSD など追加 OS 対応

---

### 実装上の注意

* PowerShell は `.ps1` 拡張子を要求するため、一時ファイル名の設計が必要
* `runner` 再帰呼び出し時は、自分自身の絶対パスで実行することを検討する
* `.run` の中身は runner が前処理し、対象 OS ブロックだけを実行対象とする
* コメント、空行、ヘッダ前後の扱いは仕様に従って処理する

---

### テスト観点

#### 正常系

* Windows ブロックのみ実行される
* Linux ブロックのみ抽出される
* macOS ブロックのみ抽出される
* 複数 OS 記述時に正しいブロックが選択される

#### 異常系

* `#script` がない
* `@end` がない
* OS ブロックが重複している
* OS ブロックが存在しない
* 対象 OS のブロックが存在しない

#### 警告系

* 未知の `@xxx` が含まれる

#### dry-run

* 対象 OS が表示される
* 実行コマンドが表示される
* 抽出後スクリプトが表示される

#### テストデータ

* `.run` ファイルを事前に用意する
* 可能であれば golden ファイルで抽出結果を比較する

---

### 進め方

1. 改善メモを確定する
2. `.run` 仕様を修正する
3. テストデータを作成する
4. 必要なら golden を用意する
5. テストコードを作成する
6. Codex に仕様とテストを渡して実装させる

---

### 一言まとめ

`.run` は共通スクリプトを作る仕組みではなく、
**OS 別スクリプトを 1 ファイルに束ね、利用者には同じ操作感を提供するための仕組み**として整理する。

---

これでそのまま **仕様書に昇格できる一歩手前の状態**。
次は「Codexに投げる用の仕様書フォーマット」に整形すると、ほぼ事故らなくなる。
